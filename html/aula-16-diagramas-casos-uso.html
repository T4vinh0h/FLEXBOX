🎯 Introdução aos Diagramas de Casos de Uso

O Diagrama de Casos de Uso atua como uma interface de tradução de alto nível que delimita o escopo funcional de uma aplicação. Ele responde exclusivamente à pergunta: O que o sistema faz frente aos seus usuários?, mantendo-se estritamente agnóstico em relação a linguagens de programação, frameworks ou infraestrutura de banco de dados.

💡 Foco: O que o sistema faz, não como ele faz.

👤 Componentes Essenciais e Suas Restrições

Ator

Representa qualquer entidade externa que interage diretamente com o sistema. Pode ser uma pessoa real desempenhando um papel específico (ex: Operador, Administrador) ou outro sistema computacional externo que consome endpoints (ex: um Gateway de Pagamento).

⚠️ Importante: Atores não fazem parte do sistema; eles residem fora de suas fronteiras.

Caso de Uso

Uma elipse que encapsula uma funcionalidade atômica e completa que gera um valor tangível para o ator.

✅ Exemplo correto: "Emitir Relatório de Vendas"

❌ Exemplo incorreto: "Clicar no Botão Salvar" (ação trivial, não um objetivo completo)

Fronteira do Sistema

Um retângulo que delimita fisicamente o escopo do desenvolvimento. Tudo o que estiver dentro do retângulo pertence ao escopo de codificação do sistema; os atores permanecem do lado de fora, cruzando a fronteira apenas através de relacionamentos.

🔗 Aprofundando os Relacionamentos Avançados

       <> (Obrigatório)
┌───────────────────────┐      ┌─────────────────────────┐
│    Sacar Dinheiro     │─────►│    Autenticar Usuário   │
└───────────────────────┘      └─────────────────────────┘
                                            ▲
       <> (Opcional)                │
┌───────────────────────┐                   │
│ Aplicar Cupom Desconto│───────────────────┘
└───────────────────────┘

Associação

Uma linha contínua que conecta um ator a um caso de uso. Ela apenas sinaliza que há uma interação comunicativa entre ambos, sem indicar fluxo de dados ou direção de precedência.

Inclusão (<>)

Define uma dependência estritamente obrigatória entre dois casos de uso. Quando o Caso de Uso Base é acionado, o Caso de Uso Incluído é executado obrigatoriamente como parte intrínseca do fluxo.

Uso: Amplamente utilizado para reaproveitar lógicas repetitivas em múltiplos lugares do sistema.

Exemplo: O caso de uso [Sacar Dinheiro] inclui obrigatoriamente o caso de uso [Autenticar Usuário]. A operação financeira é estruturalmente impossível de ser concluída sem a validação prévia de identidade.

Extensão (<>)

Estabelece um relacionamento condicional ou opcional. O caso de uso extensor só injeta seu comportamento no caso de uso base se um critério preestabelecido (conhecido como Ponto de Extensão) for ativado durante a execução.

Exemplo: O caso de uso base [Calcular Frete] pode ser estendido pelo caso de uso [Aplicar Cupom de Desconto]. A aplicação do cupom não é obrigatória para o cálculo do frete regular; ela ocorre apenas se o usuário possuir e decidir inserir um código promocional válido.

📊 Comparativo de Relacionamentos

Tipo Característica Exemplo
Associação Conexão comunicativa Usuário → Login
Include (<>) Obrigatório Sacar → Autenticar
Extend (<>) Opcional/Condicional Calcular Frete → Aplicar Cupom
