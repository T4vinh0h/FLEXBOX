📚 Introdução à Engenharia de Requisitos

A área de Engenharia de Requisitos é a "ciência" que estuda as formas de elicitar, analisar, validar e documentar as necessidades de um sistema, servindo como base para o projeto de software.

💡 Importância: É considerada um dos pontos mais críticos do desenvolvimento de software, onde ocorre a maior parte dos erros devido à falta de experiência dos clientes em expressar suas necessidades.

📋 1. Regras de Negócio (RNE)

As Regras de Negócio são normas, políticas e decisões de uma organização que definem como o sistema deve operar dentro de seu contexto. Elas são independentes da tecnologia utilizada e servem como a base para a definição dos requisitos funcionais e não funcionais.

Características:

  • Devem ser claras, objetivas e derivadas do domínio do problema
  • Por representarem decisões organizacionais, podem mudar conforme a estratégia da empresa
  • Devem ser validadas com os stakeholders

Exemplos:

  • "Um cliente só pode comprar se estiver cadastrado"
  • "A média para aprovação na instituição é 6"
  • "Clientes inadimplentes não podem realizar compras"

🔍 2. Levantamento de Requisitos (Elicitação)

O levantamento é a fase de planejamento onde se identificam e documentam as funções e restrições do software. É considerado um dos pontos mais críticos, onde ocorre a maior parte dos erros devido à falta de experiência dos clientes em expressar suas necessidades.

Processo:

  1. Elicitação: Coleta direta de informações
  2. Análise: Verificação de conflitos e viabilidade
  3. Documentação: Registro formal em documentos como a SRS (Software Requirements Specification)
  4. Validação: Revisão com o cliente

Técnicas Comuns:

Técnica Descrição Vantagens
Entrevistas A técnica mais utilizada, permitindo reformular requisitos Permite aprofundamento, mas o cliente pode se desviar do objetivo
Workshops/Mini palestras Discussões em grupo para busca de ideias e sugestões Gera brainstorming e alinhamento entre stakeholders
Observação O analista acompanha o cotidiano da empresa Entende os processos reais, não apenas o declarado
Questionários Coleta de dados de forma assíncrona Útil para grandes grupos, mas limita profundidade

⚙️ 3. Requisitos Funcionais (RF)

Os Requisitos Funcionais descrevem o que o sistema deve fazer. Eles representam as funcionalidades, tarefas e serviços que o software entrega ao usuário para satisfazer os objetivos do negócio.

Estrutura:

  • Geralmente são expressos como casos de uso
  • Utilizam verbos de ação
  • Para serem testáveis, devem conter: ID único, descrição clara, entradas, saídas e os atores envolvidos

Exemplos:

"O sistema deve listar todos os alunos cadastrados em uma turma"

"Permitir que o usuário realize login com e-mail e senha"

"O sistema deve gerar relatórios de vendas mensais"

🎯 4. Requisitos Não Funcionais (RNF)

Os Requisitos Não Funcionais definem como o sistema deve se comportar ou operar, focando em atributos de qualidade e restrições. A violação de um RNF pode tornar o sistema inutilizável (ex: lentidão extrema).

Classificação (Sommerville):

Categoria Descrição Exemplos
Requisitos de Produto Atributos de qualidade do software Desempenho (resposta em até 5 segundos), usabilidade, segurança, confiabilidade
Requisitos Organizacionais Padrões e processos da organização Padrões de implementação (ex: usar Python), requisitos operacionais
Requisitos Externos Fatores externos ao desenvolvimento Fatores legais (conformidade com LGPD), éticos

🔄 5. Exemplos Comparativos

Para distinguir os conceitos, veja como eles se relacionam no mesmo cenário:

Conceito Exemplo Característica
Regra de Negócio "Clientes inadimplentes não podem realizar compras" Política da empresa
Requisito Funcional "Gerar relatório de vendas" Uma ação do sistema
Requisito Não Funcional "O relatório deve estar disponível em formato PDF" Um modo de operação/formato

⚠️ 6. Restrições Técnicas e de Negócio

As restrições são condições obrigatórias que delimitam o escopo do projeto.

Restrições Técnicas:

Limitações de infraestrutura e tecnologia, como a capacidade de processamento dos servidores ou o uso obrigatório de uma stack tecnológica específica (ex: Windows 7 ou superior).

Restrições de Negócio:

Regras que orientam a tomada de decisão da empresa, incluindo o orçamento disponível (budget), prazos de entrega (timeboxing) e conformidade legal.

🎨 7. Prototipação

A prototipação é a criação de modelos iniciais para testar e validar ideias antes do desenvolvimento final. É uma ferramenta poderosa para alinhar a visão entre usuários e desenvolvedores.

Níveis de Fidelidade:

Nível Descrição Uso Recomendado
Baixa Fidelidade Rascunhos em papel ou diagramas simples Ideal para validar conceitos iniciais com baixo custo
Alta Fidelidade Modelos interativos e visuais que se assemelham ao projeto final Permite que os usuários contribuam na validação de requisitos funcionais e de usabilidade

Benefícios e Riscos:

✅ Benefícios: Facilita a comunicação e antecipa erros

⚠️ Riscos: Protótipos incompletos podem gerar desconfiança no cliente se não forem bem explicados
